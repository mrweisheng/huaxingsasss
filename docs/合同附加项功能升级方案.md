## 一、Context

### 1.1 业务真实诉求

合同 = 主业务金额 + 附加项金额，是一个**整体应收**。例：

```
合同 HT-2026-XXX
车价（合同主业务金额）   HKD 250,000
保险（附加项）          CNY  10,000
人工费（附加项）        CNY   5,000
其他附加项（十几项）...   CNY   5,000
─────────────────────────────────
合计应收               HKD 250,000 + CNY 20,000
```

**关键**：客户**不会**也**不需要**把每笔付款拆到具体哪一项。客户付 5 万就是 5 万——里面可能含车款 2 万 + 保险 + 人工 + 杂费十几项，业务上**不强求一一对应**。

未来用户要做的事是：**生成催款账单**——把"还欠的部分按主业务+各附加项罗列"出来给客户看。账单只看总数（已收 X 万、还欠 Y 万），不需要"X 万里有 Z 万归车价"的细分。

### 1.2 当前系统缺口

1. **没有附加项概念**：合同表只有 `total_amount`（合同字面金额），没有"应收清单上的额外项目"
2. **应收口径只是合同字面金额**：实际应收 = 合同金额 + 附加项汇总，系统无法表达
3. **超收兜底是补丁**：`ContractDetail.tsx:262-267` 的 `overpaid` 三态处理就是为这个场景打的补丁，业务概念缺失
4. **未来催款账单功能没有数据底座**：需要先有附加项数据结构，账单功能才能基于此生成

### 1.3 设计原则

1. **附加项 = 合同应收清单上的一行**，不是独立的财务实体
2. **付款不强制归属**：客户付钱不拆分，所有付款一视同仁累加到合同已收
3. **付款可选打附加项标签**：用户主动提到"这笔主要是付保险"才标，纯展示用，**不参与任何金额计算**
4. **付款 description 逻辑完全不变**：仍按现有三层来源生成（LLM 显式传入 / 凭证 receipt_data 提取 / 自动拼接兜底）
5. **超收允许**：实收 ≥ 合同应收（含附加项）即视为正常，超出部分（手续费等）算超收
6. **删除自由**：附加项随便删，付款记录上的标签自动 null，不做硬拒绝

## 二、数据模型

### 2.1 新增表 `contract_additional_items`

```sql
-- backend/sql/contract_additional_items.sql （DDL 纯 SQL，提供给用户手动执行，不走 alembic）
CREATE TABLE contract_additional_items (
    id              SERIAL PRIMARY KEY,
    contract_id     INTEGER NOT NULL REFERENCES contracts(id) ON DELETE RESTRICT,
    name            VARCHAR(200) NOT NULL,          -- 项目名称（车险 / 保养改装 / 人工费）
    amount          DECIMAL(15,2) NOT NULL,         -- 金额
    currency        VARCHAR(3) NOT NULL DEFAULT 'CNY',
    paid_to         VARCHAR(200),                   -- 付给谁（保险公司 / 修理厂）
    description     TEXT,                           -- 用途说明
    occurred_date   DATE,                           -- 发生日期，备查用
    remarks         TEXT,
    is_deleted      BOOLEAN NOT NULL DEFAULT FALSE,
    deleted_at      TIMESTAMPTZ,
    created_by      INTEGER REFERENCES users(id),
    created_at      TIMESTAMPTZ DEFAULT NOW(),
    updated_at      TIMESTAMPTZ DEFAULT NOW()
);
CREATE INDEX idx_addl_items_contract
    ON contract_additional_items(contract_id) WHERE is_deleted = FALSE;
```

**关键说明**：
- ❌ **不要 `payment_status` 字段**：附加项没有"已收/未收"概念
- ✅ `is_deleted` 软删，与项目其他表一致
- ✅ `paid_to` 字段用于对账（"我们把这笔钱付给了谁"）
- ✅ `occurred_date` 仅用于备查，不参与计算

### 2.2 `contracts` 表扩展（冗余汇总）

```sql
ALTER TABLE contracts
    ADD COLUMN additional_total_by_currency JSONB
        NOT NULL DEFAULT '{}'::jsonb;
COMMENT ON COLUMN contracts.additional_total_by_currency
    IS '附加项按币种汇总，例 {"CNY": 20000, "HKD": 500}';
```

**为何用冗余字段而非计算属性**：合同列表 / 台账分页每页 20 条，应收口径要展示"合同金额 + 附加项汇总"，避免 SUM 子查询 N+1。在 `ContractAdditionalItemService` 增删改时同步维护，与现有 `payment_service.py` 维护 `paid_amount` 同套路。

### 2.3 `payments` 表扩展（**可选标签**）

```sql
ALTER TABLE payments
    ADD COLUMN additional_item_id INTEGER
        REFERENCES contract_additional_items(id) ON DELETE SET NULL;
CREATE INDEX idx_payments_addl_item
    ON payments(additional_item_id) WHERE additional_item_id IS NOT NULL;
```

**`additional_item_id` 的语义边界（重要）**：
- 这是一个**可选的展示标签**——记录"这笔付款主要是为了某项附加项"
- **不参与任何金额聚合**：`_add_to_contract_paid` / `_add_to_contract_expense` 等所有现有逻辑**完全不动**，所有付款照常累加到 `contract.paid_amount`
- 录款时 LLM **不追问**归属，默认 null
- 用户在备注里提到"这笔含保险"时 LLM **可以**顺手设此字段，但不强求
- 删除附加项时设为 null（已在 ON DELETE SET NULL 体现）
- **唯一约束 `(contract_id, installment_number, type)` 不动**——附加项不再是独立 installment，自然不冲突

## 三、应收口径

### 3.1 合同应收 = 合同金额 + 附加项（折算到合同主币种）⚠️[v3.0 演进，详见 §十一]

> **v3.0 覆盖**：v2.0 此处为"不折算、分币种独立显示"。但 `paid_amount`（已收）早在付款录入时已折算为**合同主币种单一值**（异币种付款在 `payment_service._add_to_contract_paid` 折算进合同币种）。若附加项不折算，应收与已收口径不可比，无法支撑超收判定 / 剩余尾款 / 收款进度的自动计算——这正是 v2.0 留下 `ContractDetail` overpaid 三态补丁的根因。v3.0 起附加项也折算到合同主币种，与已收同口径可直接比较。原币种明细仍保留展示。

```
合同金额：HKD 250,000              ← contract.total_amount
附加项原币种：CNY 20,000           ← additional_total_by_currency.CNY（保留展示）
        ↕ 折算（合同签订日汇率）
附加项折算：≈ HKD 21,739           ← additional_total_in_contract_currency（与已收同币种）
─────────────────────────────────
合同应收：HKD 271,739              ← total_amount + additional_total_in_contract_currency
合同已收：HKD 50,000               ← contract.paid_amount（已折算为合同币种）
```

**折算口径**：附加项用 `contract.signed_date`（签订日）汇率折算（缺省 `date.today()`），与 `total_amount_in_cny` 折算口径一致。缺汇率时该字段为 NULL，应收口径退回 `total_amount`（前端 hero 加"含 N 项未折算"提示，避免认知割裂）。原币种明细始终展示，折算仅用于聚合到"合同应收"。

### 3.2 收款进度 ⚠️[v3.0 演进]

| 维度 | 计算 |
|---|---|
| 合同收款进度 | `paid_amount / (total_amount + additional_total_in_contract_currency)`，cap 100%（基于应收口径） |
| 单个附加项进度 | **不显示**——附加项就是清单上一行，没有单项进度概念 |

> v2.0 主业务进度用 `paid/total` 未含附加项；v3.0 统一为 `paid/receivable`，进度条 / 尾款 / 超收全部基于应收口径。缺汇率（receivable 退回 total）时等价 v2.0 行为。

### 3.3 超收

允许 `paid_amount > total_amount`，三种情形：
1. 合同金额 25 万、附加项 2 万、客户付 27 万 → 合同字面看是超收，加上附加项是刚好结清
2. 合同金额 25 万、附加项 2 万、客户付 27.5 万 → 包含附加项后还是超收，可能是手续费等
3. 现有 `ContractDetail.tsx` 的 `overpaid` 兜底**保留**，但展示文案要更新：
   - 旧文案："加项收入（实付 > 合同价，业务上常见的装饰费/过户费/议价加价）"
   - 新文案："超出合同应收（含附加项），可能是手续费或多付"

## 四、Agent 工具设计

### 4.1 新增 4 个工具（`tool_executor.py`）

| 工具 | 用途 | 注意 |
|---|---|---|
| `list_additional_items(contract_id)` | 查询合同所有附加项 + 分币种汇总 | **不返回单项已收**，附加项无收款进度概念 |
| `add_additional_item(contract_id, name, amount, currency?, paid_to?, description?, occurred_date?)` | 录入新附加项 | 录入前 LLM 必须列计划等用户确认（system prompt 软规则） |
| `update_additional_item(item_id, ...)` | 修改字段 | 同上 |
| `delete_additional_item(item_id)` | 软删除 | **不做付款引用检查**——直接软删，付款的标签字段自动 null |

### 4.2 现有工具的改动

| 工具 | 改动 |
|---|---|
| `match_and_confirm_payment` | **加可选参数 `additional_item_id`**，默认 null。LLM 在用户备注里识别到"这笔付保险"时可填，没识别到不填——**不追问** |
| `create_payment_record` | 同上 |
| `get_contract_detail` | 返回结构加 `additional_items` 列表 + `additional_total_by_currency` |

### 4.3 不需要改的现有逻辑（重要）

- ❌ **不改** `payment_service._add_to_contract_paid / _add_to_contract_expense`：所有付款照常累加到 `contract.paid_amount` / `total_expense`
- ❌ **不改** `payment_service.delete_payment` 的反扣逻辑
- ❌ **不改** `payment_service._generate_description` 与 `tool_executor._build_payment_description`：付款 description 三层来源（LLM 显式 / 凭证提取 / 自动拼接）逻辑完全不变
- ❌ **不改** `stats_service._build_kpi`：合同已收口径仍然用 `Contract.paid_amount`，因为附加项付款已经在里面了
- ❌ **不改** `payments` 表的唯一约束 `(contract_id, installment_number, type)`

**核心**：附加项付款不是独立的财务事件，它是合同总收款的一部分。所有现有金额聚合代码、description 生成代码无需任何修改。

### 4.4 提示词软规则（`prompts.py`）

按 CLAUDE.md「Agent 模式铁律」放 system prompt 软规则。

```text
## 附加项相关规则（新增段落）

1. 附加项录入：
   录入附加项前必须先列计划等用户确认：
     「将为合同 HT2026... 录入附加项：
      ① 车险（付太平洋保险）— CNY 5,000
      ② 保养改装（付XX修理厂）— CNY 5,000
      确认后我执行，请回复『确认』继续。」

2. 附加项币种检查（不自动换算）：
   录入附加项时，若币种与合同币种不一致，必须先确认：
     「合同币种是 HKD，您录入的附加项是 CNY 5,000。请确认：
      ① 是否输错了币种？
      ② 还是确实需要 CNY 计价？（系统会按 CNY 独立记账，不自动折算）」

3. 凭证录入付款（不追问归属）：
   - 客户付款不需要追问归属
   - 如果用户备注或凭证明确写了"付保险"等字样，可以在 description 里
     体现，并顺手把 additional_item_id 标到对应附加项（标签性质，不影响金额）
   - 如果看不出来，直接录付款，不追问、不标签

4. 应收口径表达：
   告知用户合同应收时使用："合同金额 HKD 25万 + 附加项 CNY 2万"分币种表达
```

## 五、前端 UI 设计

### 5.1 入口分布（关键决策）

**「添加附加项」入口只放在合同详情页**，列表 / 台账只展示金额不加交互入口。
**「添加 / 编辑」用 antd 表单 Modal**，**不**用 Agent 聊天。

| 入口 | 位置 | 形态 | 理由 |
|---|---|---|---|
| 添加附加项 | `ContractDetail` 附加项区块「+ 添加附加项」按钮 | 表单 Modal | 字段固定、单条录入，表单更直接（CLAUDE.md 铁律：纯表单 CRUD 不用 LLM） |
| 编辑附加项 | 卡片「编辑」图标 | 表单 Modal（同一组件） | 复用 add 表单 |
| 删除附加项 | 卡片「删除」图标 | `Popconfirm` | 与现有「标记完成」一致 |

**为何不在合同列表卡片上加「+ 附加项」按钮**：保持产品语言一致——「附加项是合同的细化补充」，进详情页才能录入。列表/台账卡片已经有「收/支」按钮，再加按钮会拥挤。

**为何不用 Agent 聊天弹窗**：附加项字段高度结构化（名称/金额/币种/付给谁/日期/说明），表单一目了然；Agent 聊天的优势（自然语言提取、图片 OCR）在这里不显著。后续若有"批量从图片识别附加项清单"的需求，再独立加 Agent 入口（不影响表单入口）。

**位置选择理由**：放在车辆信息条之后、付款条款之前——附加项是"对应收的细化补充"，紧跟财务概览主线；车辆信息条已是视觉分割带，附加项接续自然；付款条款是合同原始约定，附加项是合同外追加，分开放避免概念混淆。

**附加项卡片细节**：
- 不显示单项已收进度——附加项没"已收/未收"概念
- 卡片左侧 3px 色条用合同业务色（`--biz-vehicle` 或 `--biz-cross`）
- 金额数字用 `--text-primary` 黑色（不用状态色，因为附加项金额无收款状态）
- 图标按项目名启发性显示（车险 🛡️、保养 🔧、过户 📝、其他 💼），不强制

### 5.3 表单 Modal（`AdditionalItemFormModal.tsx`）

add / edit 共用同一组件，通过 `mode` prop 区分。

| 字段 | 控件 | 必填 | 说明 |
|---|---|---|---|
| 项目名称 | `Input` | ✅ | 例：车险、保养改装、过户费 |
| 金额 | `InputNumber` | ✅ | precision=2，min=0 |
| 货币 | `Select` | ✅ | CNY / HKD（默认随合同币种） |
| 付给谁 | `Input` | ❌ | 例：太平洋保险、XX修理厂 |
| 发生日期 | `DatePicker` | ❌ | 备查用 |
| 用途说明 | `Input.TextArea` | ❌ | 例：基础三责险 + 车损 |
| 备注 | `Input.TextArea` | ❌ | 业务备注 |

**币种异常提醒（前端层弱提醒，不拦截）**：
- 选择的货币与合同币种不一致时，表单底部显示黄色提示条：「附加项币种与合同币种不一致，系统将按所选币种独立记账，不自动折算汇率」
- 不弹确认框，不阻塞提交——用户的录入意图优先

### 5.4 列表 / 台账（仅展示，无交互）⚠️[v3.0 演进]

合同列表 / 台账行金额区显示「合同金额 + 附加项折算角标」（折算到合同币种，可与主数字相加理解）：

```
HT-2026-XXX  XX汽车贸易公司
车辆业务 · 进行中
─────────────────────────────
HKD 250,000  +HKD 21,739   ← 合同金额 + 附加项折算（hover 显原币种 CNY 20,000）
已收 HKD 50,000  ·  剩余 ...
```

> v2.0 为分币种角标（HKD 250,000 + CNY 20,000）；v3.0 改用折算值角标（`var(--brand-gold)`），tooltip 保留原币种明细。无附加项合同退化为单行金额。卡片不加「+ 附加项」按钮。

### 5.5 录款 Modal（`ReceiptChatModal.tsx`）

**保持不变**——不加归属选择器。LLM 自主决定是否标 `additional_item_id`（识别到用户提及才标，否则 null）。

### 5.6 颜色规范

按现有 CSS 变量（`frontend/src/pages/ContractList.css:12-30`）：
- 业务色（车辆 `--biz-vehicle` #2d5b8a / 两地牌 `--biz-cross` #b8423b）：附加项卡片左侧色条
- 「+ 添加附加项」按钮：`type="primary"` 用品牌色 `--brand-primary`，与「+ 新建合同」一致
- 付款标签：用 `--brand-gold` #c9952b 浅底
- 不引入新业务色——避免色彩污染

## 九、关键设计决策（已与产品方确认）

| 决策点 | 选择 | 理由 |
|---|---|---|
| 颗粒度 | 逐项记录 | 支持后续按类型统计、利润分析 |
| 与支出（expense）的关系 | 完全独立 | 附加项是"应收清单"，公司付出去的钱仍走现有 expense 流程 |
| 客户付款归属 | **不强制归属，可选标签** | 客户付钱不会拆分到具体哪一项，硬归属与业务直觉相悖 |
| LLM 凭证录付款 | **不追问归属** | 看用户备注顺手标，识别不到就空着 |
| 付款金额聚合 | **一视同仁累加 `paid_amount`** | 附加项付款是合同总收款的一部分，不是独立财务事件 |
| 付款 description | **现有三层逻辑完全不变** | LLM 显式 / 凭证提取 / 自动拼接兜底 |
| 附加项币种 | 原币种存储 + 聚合时折算到合同币种 ⚠️[v3.0] | 折算用签订日汇率，与已收同口径可比；原币种仍展示（见 §十一） |
| 币种不一致处理 | 表单层弱提醒，不拦截 | 用户录入意图优先 |
| 应收汇总展示 | 折算到合同主币种统一口径 ⚠️[v3.0] | 与 paid_amount 同币种可比，支撑超收/进度/尾款计算（见 §十一） |
| 是否允许超收 | 允许（警告不拦截） | 与现有 overpaid 处理一致；客户付费可能含手续费等 |
| 删除附加项 | **直接软删，无硬保护** | 业务上不强归属，付款标签自动 null 即可 |
| 前端添加入口形态 | **antd 表单 Modal**，不用 Agent 聊天 | 字段固定结构化，表单更直接（CLAUDE.md 铁律：纯表单 CRUD 不用 LLM） |
| 添加入口位置 | **只在详情页**，列表/台账不放 | 保持产品语言"附加项是合同的细化补充"，列表卡片不再加按钮 |

## 十、风险与未决项

### 风险

| 风险点 | 影响 | 缓解措施 |
|---|---|---|
| `additional_total_by_currency` JSONB 与明细表不一致 | 列表显示错乱 | Service 层所有变更都在事务内一起提交；后续可加数据校验脚本 |
| 历史合同已存在的"超收"金额无法迁移到附加项 | 历史数据应收口径不准 | 上线后管理员手动补录历史附加项；`overpaid` 兜底文案保留 |
| ⚠️[v3.0] 折算字段需回填 | DDL 加列后已有附加项合同的 `additional_total_in_contract_currency` 为 NULL | 执行 §十一.3 回填命令遍历合同重算；之后增删改附加项自动维护 |
| LLM 误标 `additional_item_id` | 付款标签错位 | 标签不影响金额计算；前端付款编辑允许人工修正标签 |

2. 附加项是否需要"分类"枚举字段（车险 / 保养 / 改装 / 其他）做分组统计——当前 `name` 自由文本足够，未来真有利润分析需求再加

## 十一、应收口径统一演进（v3.0）

> 2026-06-14 追加。基于 v2.0 附加项数据底座，把"合同应收 = 合同金额 + 附加项"升级为**全系统统一口径**，并修复详情接口软删泄漏（审核 P1）。本节是 v3.0 的权威说明，覆盖正文中标注 ⚠️[v3.0] 的 v2.0 决策。

### 11.1 动机

v2.0 附加项分币种不折算，与已收（合同主币种单一值）口径不可比，导致超收/剩余/进度只能基于 `total_amount`——客户付清"车款+保险"会被误判超收。统一为折算口径后，应收与已收同币种可直接比较，overpaid 三态补丁被真正解决。

### 11.2 改动清单

| 层 | 改动 |
|---|---|
| DDL | `contracts` 加 `additional_total_in_contract_currency DECIMAL(15,2)`（NULL=未折算/缺汇率） |
| Model / Schema | 加对应字段（Optional，向后兼容扩展） |
| Service | `_recalculate_contract_summary` 增删改附加项时一并折算写入该字段（try/except 兜底为 NULL，不阻断增删改） |
| 详情接口 P1 修复 | `get_contract_detail` 加 `with_loader_criteria(ContractAdditionalItem, is_deleted==False, include_lazy=True)`，selectinload 与 lazy load 都过滤软删行（修审核 P1 泄漏） |
| Agent 工具 | `get_contract_detail` 返回 `additional_total_in_contract_currency` |
| 提示词 | 「合同附加项」第 4 条改统一口径 + 缺汇率 null 降级说明（null 不当 0 相加） |
| 前端详情 | hero 改"合同应收"口径（total+折算），overpaid/unpaid/progress/paymentState 全部基于 receivable；缺汇率退回总额 + "含 N 项未折算"提示 |
| 前端列表 / 台账 | 角标用折算值 + `var(--brand-gold)`，tooltip 保留原币种明细 |

**未改动（零侵入）**：`payment_service` 聚合（`_add_to_contract_paid` 等）、`stats_service`、付款 description 三层来源、payments 唯一约束——全部不动。

### 11.3 部署步骤（DDL + 回填）

```bash
# 1. 执行 DDL（幂等，重跑整文件即可；第 4 段为 v3.0 新增）
backend/sql/contract_additional_items.sql

# 2. 回填已有附加项合同（DDL 后这些合同新字段为 NULL）
cd backend && PYTHONIOENCODING=utf-8 uv run python -c "from app.db.session import SessionLocal; from app.models.contract import Contract; from app.services.contract_additional_item_service import AdditionalItemService; db=SessionLocal(); ids=[r[0] for r in db.query(Contract.id).filter(Contract.is_deleted==False).all()]; [AdditionalItemService._recalculate_contract_summary(db, c, write_back=True) for c in ids]; db.commit(); print(f'refilled {len(ids)} contracts'); db.close()"
```

回填后增删改附加项自动维护该字段，无需再跑。缺汇率的合同字段保持 NULL（前端 / Agent 降级为合同金额 + 分币种明细）。

### 11.4 折算口径对称性（重要，避免误读为"完全统一"）

- **应收侧**：`total_amount_in_cny` 与 `additional_total_in_contract_currency` 均用**合同签订日**（`signed_date`）汇率折算
- **已收侧**：`paid_amount` 用**付款发生日**（每笔 `paid_date`）汇率折算
- 两侧各自口径对称——应收侧一口径、已收侧一口径，符合项目既有折算惯例（非"全合同单日汇率"）。同一合同在汇率波动期，应收与已收的"合同币种值"用的是不同时点汇率，属可接受近似。

### 11.5 边界与降级

| 场景 | 行为 |
|---|---|
| 同币种附加项 | 直接相加，无折算误差 |
| 异币种附加项 + 有汇率 | 折算到合同币种，写入 `additional_total_in_contract_currency` |
| 异币种附加项 + 缺汇率 | 字段为 NULL；前端 hero 退回"合同总额" + "含 N 项未折算"提示；Agent 用分币种 `additional_total_by_currency` 表达，不把 null 当 0 |
| 删除附加项（P1） | `with_loader_criteria` 过滤软删行，详情接口不再泄漏；汇总字段同步重算 |
