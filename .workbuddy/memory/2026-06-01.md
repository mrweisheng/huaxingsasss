## Agent 聊天助手 - 图片上传后无回复 Bug 修复

### 问题现象
用户在前端聊天助手上传图片后，LLM 调用 `analyze_image` 工具成功返回结果，但前端界面停留在"好的，我来分析这张图片"不再继续。

### 根因分析
**agent.py ReAct 循环中，第二轮 LLM 调用（工具结果返回后）可能返回空文本，原因：**
1. 用户消息为自动生成的"请分析上传的图片内容，提取关键信息并总结"，但系统 prompt 要求凭证走 `match_receipt` 流程
2. LLM 在"总结"和"继续调 match_receipt"之间犹豫，最终返回空文本
3. `full_text=""`、`tool_calls=[]` → 代码分支到"无工具调用"→ 保存空内容并 yield done
4. 前端收到 done 事件但无新文本可显示 → 界面"没有下文"

### 修改内容（agent.py）
1. **每轮 ReAct 迭代增加 logging**：`ReAct迭代 #N: messages=X 条` 和 `ReAct迭代 #N 完成: full_text_len=X, tool_calls=X`
2. **LLM 调用增加 try/except**：捕获异常后尝试用工具结果生成兜底回复
3. **新增 `_build_tool_result_fallback`**：从最近的 tool 消息中提取结构化数据（凭证/合同/文档），组装为可读的自然语言回复
4. **空文本检测**：第二轮及以上若 LLM 返回空文本且无工具调用，自动触发兜底回复

### 涉及文件
- `D:\CODE PROJECT\huaxingsasss\backend\app\ai\agent.py` — ReAct 循环 + 新增 `_build_tool_result_fallback` 方法

## 凭证路径双保险 - 代码审核后的加固修复

### 问题 1：DB 兜底路径可被重复消费
`_get_receipt_image_path` 第 3 层（DB 查询）没有消费标记，同一会话里如果多次创建付款，会重复关联同一张图片。修复：在 `ChatHistory.extra_metadata` 中加 `receipt_consumed: true` 标记，查询时跳过已消费记录。

### 问题 2：临时文件不存在时写入无效路径
`_ensure_file_in_receipt_dir` 在文件不存在或复制失败时原本返回原路径（如 `agent_upload/xxx`），这个路径实际不存在，会导致付款记录关联了一个虚假路径。修复：改为返回 `None`。`update_payment` 中检测到 `None` 时删除 `receipt_image_path` key，避免用空值覆盖已有路径。

### 修改内容（tools.py）
1. `_get_receipt_image_path`：DB 兜底返回路径前，检查并写入 `receipt_consumed` 标记
2. `_ensure_file_in_receipt_dir`：文件不存在和复制异常时返回 `None` 而非原路径
3. `update_payment`：解析结果为 `None` 时移除 `receipt_image_path` key
4. 类型注解 `_ensure_file_in_receipt_dir` 返回值改为 `Optional[str]`

### 设计原则

3. `_add_to_contract_expense`：CNY 字段改为有条件更新
4. `update_payment`：保持 None 语义传递给累加方法

**`tools.py`** — 工具定义：

2. `create_expense`：删除 currency 的 `"default": "CNY"`

**`prompts.py`** — 提示词：

## 凭证关联后状态卡在 pending 的 Bug 修复

### 问题现象
用户上传收据图片，AI 成功分析（analyze_image）并匹配（match_receipt）到对应的 pending 付款记录，然后调用 update_payment 关联。但关联后：
1. 付款状态仍为 pending，没有自动转为 paid
2. 前端看不到上传的收据凭证图片

### 根因分析——两条链路同时断裂

**链路一：状态转换门控只看 receipt_image_path**

`PaymentService.update_payment` 中判断"是否有凭证"的代码：
```python
had_receipt = bool(payment.receipt_image_path)
now_has_receipt = bool(payment.receipt_image_path)
if was_pending and not had_receipt and now_has_receipt:
    payment.status = 'paid'
```
只检查 `receipt_image_path`（图片文件路径），不检查 `receipt_data`（OCR 凭证数据）。

**链路二：receipt_image_path 解析可能失败**

跨对话回合时（analyze_image 在上一轮，update_payment 在下一轮），`_pending_receipt_path` 缓存已清空，只能走 DB 兜底。DB 兜底能找到 `analyze_image` 记录并返回 `agent_upload/{file_id}`，但 `_ensure_file_in_receipt_dir` 查找 temp 文件时可能因文件已清理而返回 None，导致 `receipt_image_path` 被静默移除。

**致命组合**：
- DB 兜底找到历史记录 → 返回 `file_path`
- temp 文件已清理 → `_ensure_file_in_receipt_dir` 返回 None
- `receipt_image_path` 从 updates 中移除
- `receipt_data` 仍然在 updates 中（OCR 数据成功写入）
- 但状态转换只看 `receipt_image_path` → 永远是 False → 永不触发

同时，DB 兜底还先标记了 `receipt_consumed = True`，导致 retry 也被阻止。

### 修改内容

**`payment_service.py`**：
1. `had_receipt` 判断改为 `bool(receipt_image_path) or bool(receipt_data)`
2. `now_has_receipt` 同理
3. 新增兜底分支：如果 `status == 'pending'` 但凭证证据已存在（历史遗留），强制转换

**`tools.py`**：
1. `_get_receipt_image_path` DB 兜底：移除 `receipt_consumed` 标记机制（提前标记导致 retry 被阻）
2. 更新函数注释

### 设计原则
`receipt_data`（AI 从收据中提取的结构化数据：付款人、金额、币种、日期等）本身即是凭证证据。图片文件可能因 temp 清理而丢失，但数据不会。两者任一存在即视为有凭证。

## analyze_image 立即持久化文件——根治凭证丢失

### 问题
之前 `analyze_image` 分析成功后只在 `_pending_receipt_path` 和返回的 `file_path` 中写 `agent_upload/{file_id}`（temp 路径），文件复制到永久凭证目录要等到 `update_payment` 调用 `_ensure_file_in_receipt_dir` 时才执行。跨对话回合时 temp 文件被清理，复制失败，`receipt_image_path` 为空 → 状态不转 + 前端看不到原图。

### 修复
在 `analyze_image` 每个成功分支中，**分析完成后立即调用 `_ensure_file_in_receipt_dir` 将文件从 temp 复制到永久凭证目录**，返回永久路径（如 `2026/06/xxx.jpg`）。文件几毫秒前刚被成功读取，不可能不存在。

涉及的 6 个分支：
1. 图片 VL 分析（收据/合同主流程）
2. PDF 文本解析（DeepSeek 文本模型）
3. PDF VL 兜底（扫描件无文本时）
4. PDF general 分析
5. Word/文本文件分析
6. Excel 文件分析

### 效果
- `_pending_receipt_path` 和 ChatHistory 中存储的是永久路径而非 temp 路径
- `update_payment` 时 `_ensure_file_in_receipt_dir` 收到永久路径直接通过（不匹配 `agent_upload/` 前缀）
- 凭证文件在分析时就已经在永久目录中，不会再被清理
- 新增的每一个凭证都能在前端看到原图

### 涉及文件
- `backend/app/ai/tools.py` — `analyze_image` 方法 6 个返回分支
