# 修复 Agent 确认循环 + 会话历史混乱问题

## 问题概述

用户上传 PDF 合同后，Agent 反复要求确认 4-5 次才能完成录入，且历史会话记录混乱。

## 根因分析

通过逐条分析聊天记录 + explore agent 代码分析，定位了 7 个独立问题：

### 问题 1：确认检查方向错误（导致第 1 次拦截）

`unified_agent.py:165-177` 的 `_has_confirmation_in_context()` 检查 `messages[-1]`（LLM 含 tool_calls 的回复）的文本是否包含确认关键词。

**实际流程**：
1. LLM 展示计划，问"是否确认？" → 用户说"OK"
2. LLM 回复"好的，开始录入！" + tool_calls=[create_customer]
3. 检查 `messages[-1]` 的文本 "好的，开始录入！" → 不含"确认/是否/同意"等关键词 → **被拦截**
4. LLM 被迫重新展示详细计划

**根因**：检查方向错误——应检查用户消息（HumanMessage）是否包含确认关键词，而非检查 LLM 自己的回复。

### 问题 2：确认关键词不完整（导致"OK"无效）

`_CONFIRM_KEYWORDS` 缺少常见肯定回复：
- `"OK"` / `"ok"` → 不命中
- `"可以"`（不带"吗"）→ 不命中（只有"可以吗"）
- `"好"` / `"行"`（单字）→ 不命中

用户说"OK"后，即使改为检查用户消息，"OK"也不在关键词列表中。

### 问题 3：LLM 为每个工具单独确认（导致合同创建循环）

Prompt 中 `prompts_v2.py:53-58`：
> "所有写入操作（create_customer / create_contract / ...）前，必须先向用户展示操作计划并请求确认"

LLM 将此解读为：create_customer 和 create_contract 是两个独立工具，各需单独确认。用户确认整体计划后，create_customer 执行成功，但 LLM 仍为 create_contract 重新展示计划并请求确认。

### 问题 4（P0）：finalize_node 消息无限复制

`unified_agent.py:402-452` — `finalize_node` 遍历 `state.get("messages", [])` 写入 chat_history。由于 `messages` 有 `add_messages` reducer，LangGraph checkpointer 恢复旧状态后累积所有历史消息。`_finalized` 无 reducer，每次被 initial_state 覆盖为 False → 每轮都全量写入 → 第 N 轮时前 N-1 轮消息被重复写入 N-1 次。

**SQL 铁证**：同一 tool_call_id 出现在 25 条 chat_history 记录中。

### 问题 5（P1）：tool_calls 字段格式不匹配

后端存 `args: dict`（LangChain 格式），前端期望 `arguments: string`（`frontend/src/types/agent.ts:31-37`）。历史加载时 arguments 为 undefined。

### 问题 6（P2）：历史消息丢失 attachments/thoughts

`agent.py:247-267` 的 `get_history` 不返回这两个字段，`frontend/src/services/agent.ts:21-31` 的 `mapMessage` 也不映射。用户上传文件的对话历史回看时附件消失。

### 问题 7（P3）：tool summary 不持久化

`extract_tool_summary()` 通过 SSE 发送但未写入 DB，历史回看时 ToolCallBlock 只显示"已获取数据"占位文本。

**实际流程**：
1. 用户确认整体计划 → create_customer 执行成功
2. LLM 生成文本 "客户已创建成功，接下来创建合同，请确认"（无 tool_calls）→ 终结
3. 用户说"确认" → LLM 仍生成文本重新展示计划（不调 create_contract）
4. 用户再次说"确认" → LLM 仍然重新展示
5. 直到第 3 次"确认"，LLM 才真正调用 create_contract

**根因**：提示词未明确"用户已确认整体计划后，后续写入工具直接执行"。

## 修复方案

### 修复 1：修改确认检查逻辑

**文件**：`backend/app/ai/orchestrator/unified_agent.py:165-177`

将检查目标从 LLM 回复改为用户消息：

```python
def _has_confirmation_in_context(messages: list) -> bool:
    """检查用户是否在最近的回复中包含确认关键词。"""
    if not messages:
        return False
    # 从后往前找最近的 HumanMessage
    for msg in reversed(messages):
        if isinstance(msg, HumanMessage):
            content = (msg.content or "").lower()
            return any(kw in content for kw in _CONFIRM_KEYWORDS)
        # 如果先遇到无 tool_calls 的 AIMessage，说明 LLM 在等待用户回复
        if isinstance(msg, AIMessage) and not getattr(msg, "tool_calls", None):
            return False
    return False
```

### 修复 2：扩展确认关键词

```python
_CONFIRM_KEYWORDS = frozenset({
    "确认", "是否", "同意", "继续", "对吗", "正确吗",
    "确认吗", "可以吗", "行吗",
    # 新增常见肯定回复
    "ok", "好的", "好", "可以", "行", "没问题",
    "对", "是的", "执行", "录入", "创建",
})
```

### 修复 3：优化提示词减少重复确认

**文件**：`backend/app/ai/prompts_v2.py:53-58`

改为：
```
### 确认规则（重要）
- 用户确认整体计划后，后续在同一业务流程中的写入工具直接执行，不要为每个工具单独确认。
- 仅在首次展示操作计划时请求确认。客户创建成功后，合同创建直接执行，无需再次确认。
- 收到拒绝/修改要求时调整计划再确认。
```

### 修复 4（P0）：finalize_node 只保存当前轮新增消息

**文件**：`backend/app/ai/orchestrator/unified_agent.py:402-452`

方案 A（推荐）：在 `AgentState` 中新增 `_last_saved_msg_count: int` 字段（无 reducer），`finalize_node` 只写 `messages[_last_saved_msg_count:]`，写入后更新该字段。

方案 B：在 `initial_state` 中记录当前轮的起始 index，`finalize_node` 只写该 index 之后的消息。

### 修复 5（P1）：get_history 中 tool_calls 格式映射

**文件**：`backend/app/ai/agent.py:255-263`

在合并 tool_results 时，将 LangChain 的 `args` 字段映射为前端期望的 `arguments` 字段：
```python
if "args" in tc_copy and "arguments" not in tc_copy:
    tc_copy["arguments"] = json.dumps(tc_copy.pop("args"), ensure_ascii=False)
```

### 修复 6（P2）：get_history 返回 attachments 和 thoughts

**文件**：`backend/app/ai/agent.py:247-267`

在 `ChatHistory` 模型中已有 `extra_metadata` JSON 字段。需要：
1. `finalize_node` 写入时将 attachments/thoughts 存入 `extra_metadata`
2. `get_history` 返回时从 `extra_metadata` 提取并返回
3. 前端 `mapMessage` 映射这两个字段

### 修复 7（P3）：summary 持久化

**文件**：`backend/app/ai/orchestrator/unified_agent.py` + `backend/app/ai/agent.py`

在 `finalize_node` 中，将 `extract_tool_summary()` 的结果存入对应 tool role 记录的 `extra_metadata`。`get_history` 返回 assistant 消息时，从 tool 记录中提取 summary 并附加到 tool_calls 中。

## 验证方式

### 确认循环修复验证
1. 本地启动后端：`cd backend && uv run uvicorn app.main:app --reload`
2. 上传 PDF 合同，说"录入合同"
3. Agent 展示计划后，用户说"OK"
4. 验证 Agent 直接执行创建客户 → 创建合同，不再重复确认
5. 检查 chat_history 表确认对话记录精简

### 历史会话修复验证
1. 触发一个包含多个工具调用的对话（如录入合同）
2. 加载历史会话，验证消息不重复
3. 验证工具调用的 arguments 字段正常显示
4. 验证附件信息在历史回看时可见
5. 验证工具摘要显示有意义的文本而非占位符
